
# خطة إصلاح مشكلة القاعات المكررة

## تشخيص المشكلة

### السبب الجذري:
عند إنشاء جدول `rooms` في البداية، تم إضافة قيد `UNIQUE` على عمود `name` فقط:

```sql
CREATE TABLE public.rooms (
    id UUID NOT NULL DEFAULT gen_random_uuid() PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,  -- ← هذا القيد يمنع أي اسم مكرر على مستوى النظام بالكامل
    type room_type NOT NULL DEFAULT 'lecture',
    ...
);
```

هذا يعني أن:
- إذا أضاف المستخدم "أ" قاعة باسم "قاعة 101"
- لا يستطيع المستخدم "ب" إضافة قاعة بنفس الاسم "قاعة 101"
- رغم أن كل مستخدم يجب أن يكون لديه بياناته الخاصة والمعزولة

### المطلوب:
تغيير القيد ليكون على مستوى المستخدم: `UNIQUE (user_id, name)` بدلاً من `UNIQUE (name)`

---

## الحل المقترح

### Migration SQL لإصلاح المشكلة:

```sql
-- 1. حذف القيد القديم (العام)
ALTER TABLE public.rooms DROP CONSTRAINT IF EXISTS rooms_name_key;

-- 2. إضافة قيد جديد على مستوى المستخدم
ALTER TABLE public.rooms 
ADD CONSTRAINT unique_user_room_name UNIQUE (user_id, name);
```

## كيف سيعمل بعد التعديل

### قبل التعديل:
```
المستخدم أ: يضيف "قاعة 101" ✅
المستخدم ب: يحاول إضافة "قاعة 101" ❌ (خطأ: القاعة موجودة)
```

### بعد التعديل:
```
المستخدم أ: يضيف "قاعة 101" ✅
المستخدم ب: يضيف "قاعة 101" ✅ (مسموح لأنها تنتمي لمستخدم مختلف)
المستخدم أ: يحاول إضافة "قاعة 101" مرة أخرى ❌ (لا يمكن لأنها موجودة لديه بالفعل)
```

## الملفات المتأثرة

| الملف | التغيير |
|-------|---------|
| Migration جديد | حذف القيد القديم وإضافة القيد الجديد |

## ملاحظة مهمة

هذا التغيير يتطلب فقط تعديل على قاعدة البيانات ولا يحتاج أي تعديلات على الكود الأمامي (Frontend).
